# Goal 04C Audit - Unified Multi-Venue Desktop Market Experience

Date: 2026-06-03

Result: **CONDITIONAL_PASS**

## Scope

Audited the Goal 04C desktop market-data workflow across:

- unified market search and selection in `apps/desktop`;
- renderer-to-Tauri market-data command boundary;
- Rust/Tauri read-only provider command handlers;
- desktop state model, safety surfaces, and tests;
- active market-data, desktop, and Tauri security documentation.

This audit did not use archived or superseded implementation context.

## Findings Fixed

### Empty search could invoke provider discovery

Before the audit fix, an empty or whitespace-only market search could still enter
the command client and request provider discovery. That was not a live-trading
risk, but it was unnecessary external API use and an ambiguous desktop state.

Fix:

- `runMarketSearch` and `beginMarketSearch` now normalize queries and return the
  initial disconnected search state for empty input.
- The Tauri `market_search` command now also returns a secret-free disconnected
  no-request response for empty input.
- Desktop workflow tests now verify that empty search does not call provider
  commands.

### Venue health conflated REST snapshot availability with stream availability

Before the audit fix, a selected venue could show `unavailable` or
`credentials-required` in the venue connection field when the normalized
order-book snapshot was fresh but the WebSocket subscription path was unavailable
or credential-gated. That made real snapshot data look unavailable.

Fix:

- Selected venue `connectionStatus` now reflects the selected market-data
  snapshot state.
- Stream/WebSocket status remains visible separately in the ladder header,
  bottom strip, and selected venue summary.
- Desktop workflow tests now verify that a fresh Kalshi snapshot can show
  connected data while the stream remains `credentials-required`.

### Tauri-side provider requests had no explicit timeout

Before the audit fix, Rust/Tauri provider HTTP requests used the default reqwest
client timeout behavior. A stalled provider request could make the desktop search
or order-book command feel hung.

Fix:

- The Tauri read-only HTTP client now has a 12-second timeout.
- Security/config tests assert the timeout remains present in the Tauri command
  source until real Rust compile checks are available in the environment.

### CSP documentation implied renderer provider networking

The active Tauri security spec still described a provider-aware renderer
`connect-src` allowlist. Goal 04C intentionally keeps provider networking inside
Rust/Tauri commands, so the actual renderer CSP correctly remains `connect-src
'self'`.

Fix:

- `docs/architecture/TAURI_SECURITY_AND_COMMAND_SPEC.md` now distinguishes the
  current renderer CSP from any later ADR that might move provider networking
  into the renderer.

## Remaining Non-Blocking Debt

- `cargo` is not installed in the current Windows/Codex environment, so
  `apps/desktop/src-tauri/src/main.rs` still needs a real `cargo check` or
  `pnpm desktop:dev` verification after Rust/MSVC tooling is installed.
- The Rust/Tauri command path duplicates some provider normalization already
  covered by the TypeScript market-data adapters. This is acceptable for the
  current vertical slice but should be revisited when persistent Tauri-side
  provider runtime/session management is added.
- Polymarket WebSocket normalization exists in `packages/market-data`, but the
  desktop command path still returns `unavailable` for live stream subscription
  until a safe persistent Tauri-side WebSocket transport/session is implemented.
- Kalshi WebSocket remains `credentials-required` until a local credential
  provider supplies official authenticated handshake headers.

## Live and Safety Review

- No live order placement, cancellation, account mutation, provider credential,
  auth header, signing payload, geoblock workaround, bot, strategy, copy trading,
  or AI signal path was introduced.
- Renderer provider API access remains blocked.
- Renderer filesystem, shell/process, provider SDK, and credential access remain
  blocked by the app model and Tauri capability tests.
- Execution mode stays disabled, one-click stays off, kill switch stays visible,
  account metrics remain unknown, and live gates remain blocked.

  `apps/desktop/src-tauri`.

## Decision

Goal 04C is ready to hand off to Goal 05. Native Tauri/Rust compile
verification now passes after installing Rust/Cargo and Microsoft C++ Build
Tools on Windows. Node.js was upgraded to `22.22.3`, satisfying the repo engine
range and Vite's preferred `20.19+` or `22.12+` floor.
